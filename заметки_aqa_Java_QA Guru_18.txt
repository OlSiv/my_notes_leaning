КАК СОЗДАТЬ SSH-ключ для GitHub для Windows - 

1. Генерация SSH-ключа
Откройте терминал: Запустите PowerShell или Git Bash.
Сгенерируйте ключ: Введите следующую команду, заменив ваш_email@example.com на свой адрес электронной почты:
--
ssh-keygen -t rsa -b 4096 -C "ваш_email@example.com"
----
Выберите место сохранения: Когда появится запрос на ввод имени файла, нажмите Enter, чтобы использовать путь по умолчанию (C:\Users\YourUsername\.ssh\id_rsa).
Установите пароль (опционально): Вам будет предложено ввести пароль (passphrase). Вы можете нажать Enter, чтобы пропустить этот шаг, но для дополнительной безопасности рекомендуется установить пароль.
Подтвердите пароль: Если вы задали пароль, введите его снова, чтобы подтвердить.
Проверьте файлы: После завершения в папке C:\Users\YourUsername\.ssh\ появятся два файла: приватный ключ id_rsa и публичный ключ id_rsa.pub. 

2. Добавление публичного ключа на GitHub
Откройте настройки GitHub: Перейдите в свой профиль в правом верхнем углу, затем выберите Settings и слева выберите SSH and GPG keys.
Создайте новый ключ: Нажмите кнопку New SSH Key.
Скопируйте содержимое публичного ключа: Откройте файл id_rsa.pub в любом текстовом редакторе (например, Блокнот) и скопируйте всё его содержимое, которое начинается с ssh-rsa и заканчивается вашим email.
Вставьте ключ на GitHub: Вставьте скопированный ключ в поле Key на GitHub.
Добавьте ключ: Назовите ключ в поле Title (например, "Windows-ноутбук") и нажмите кнопку Add SSH Key.

origin - это слово указывает, что мы обращаемся к удаленному репозиторию 

git push -u origin main 
(-u - нужно делать только в первый раз, когда в удаленном репозитории еще нет ветки, которую мы отправляем)

gradle - это фреймворк, который осуществляет компиляцию и сборку проекта, управление зависимостями проекта и т.п. (собирает кучу кода в то, что можно запустить как программу)

в ветке main (master) хранится код, который прошел все проверки и валидации и он гарантированно рабочий 

git branch 
(посмотреть все ветки)

git branch new-branch
(создаст новую ветку new-branch)

git checkout new-branch
(переключит на ветку new-branch)

pull request - отправка запроса на вливание своей ветки с изменениями в ветку main (master)

тот кто принимает изменения, смотрит их на github во вкладке Pull requests 

git pull
(стягивает изменения с удаленного репозитория, с этой команды должна начинаться работа)

git merge main
(находясь в другой ветке, к примеру my_branch, слить эту ветку с веткой main)

конфликты git 

конфликты всегда нужно разбирать самому 

конфликты можно разбирать долго, иногда целыми днями 

в консоли резолвить конфликты не очень удобно 

rebase - изменяет историю коммитов, а merge - не изменяет, лучше делать merge 

fetch - похож на git pull, но загружает изменения не только одной ветки, а все изменения с удаленного репозитория
